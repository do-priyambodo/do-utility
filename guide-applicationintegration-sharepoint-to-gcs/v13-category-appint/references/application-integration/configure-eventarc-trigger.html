This page describes how to configure an {{eventarc_name_short}} trigger in {{integration}}.

An {{eventarc_name_short}} trigger lets you run integrations based on events routed through {{eventarc_standard_name}}. Using an {{eventarc_name_short}} trigger, you can build integration flows triggered by events from various {{gcp_name}} services, third-party services, and custom event sources. For example, you can use the trigger to initiate an integration when a new object is uploaded to a {{storage_name}} bucket.

In {{console_name_article}}, you can create and configure an {{eventarc_name_short}} trigger in the following ways:

  • Using {{integration}}
  • Using {{eventarc_name_short}}

Before you begin

Before configuring the {{eventarc_name_short}} trigger, enable the {{eventarc_api}} and the {{pubsub_api}}:

Considerations

Consider the following when configuring an {{eventarc_name_short}} trigger:

  • If creating the {{eventarc_name_short}} trigger fails during integration publication due to configuration errors, your integration will still publish, but the trigger won't be active.

    To fix this, unpublish the integration, carefully review and correct the {{eventarc_name_short}} trigger configuration, and then republish the integration. You can check the {{eventarc_name_short}} trigger creation status by clicking notifications_none (Notification) in the {{console}} tool bar.

  • Publishing a new version of your integration with a modified {{eventarc_name_short}} trigger configuration automatically replaces the old trigger with a new one. This ensures your integration always uses the latest trigger configuration and deletes any obsolete triggers.
  • If you publish a new version of your integration and use the same {{eventarc_name_short}} trigger and its configuration, a new trigger isn't created. Instead, the existing {{eventarc_name_short}} trigger is reused for your new integration version.

  1. Drag and drop the {{eventarc_name_short}} element in the integration editor.

Configure the {{eventarc_name_short}} trigger

To configure an {{eventarc_name_short}} trigger, click the {{eventarc_name_short}} Trigger element in the integration editor. The {{eventarc_name_short}} Trigger configuration pane appears showing the following fields:

  • Label: (optional) the visible name of the {{eventarc_name_short}} trigger in your integration. You can change the label to a name of your choice.
  • Notes: (optional) you can add notes to provide additional information about the trigger.
  • Click Configure Eventarc trigger and provide the following inputs in the Eventarc trigger editor:
    • Trigger name: enter a name for the trigger.
    • Trigger type: select one of the following trigger types:
      • Google source: select this option to create a trigger for Pub/Sub, Cloud Storage, Firestore, and other Google event providers.
      • Custom: select this option to create a trigger for a custom event source.
      • Third-party: select this option to create a trigger for non-Google providers that offer an Eventarc source. For more information, see Third-party events in Eventarc.
    • Event provider: select the event provider for the trigger. The list of event providers depends on the trigger type. For more information, see Event providers and destinations.
    • Event type: select the event type for the trigger. Your trigger configuration varies based on the event provider you select. For more information, see Event types.
    • Region: select the region for the trigger.
      Tip: For optimal performance and to minimize latency, ensure that your {{eventarc_name_short}} trigger, the monitored {{gcp_name}} resource, and your service are all located in the same {{gcp_name_short}} region.
    • Filters: (optional) add filters to select the APIs that match your criteria. For example, if you filter the APIs by a specific name, the event is triggered only if the APIs matching the filter criteria are updated.
    • Service account: select the service account to use for the trigger. If you don't select a service account, the trigger uses the {{compute_name}} default service account as its identity when invoking your service. The service account associated with the trigger must have the required permissions to invoke your service.

    • Retry on failure: select to enable trigger retries if the destination fails to process the event.
    • Network attachment: select the network attachment to use for the trigger. This list is generated based on the network attachments created in your project.

      Network attachments facilitate communication between service producer {{vpc_name_first_mention}} networks and consumer {{vpc_name_short}} networks. {{eventarc_name_short}} uses the configured network attachment to establish a connection to an internal HTTP endpoint hosted within the consumer's {{vpc_name_short}} network, enabling delivery of event data. For more information about creating and managing network attachments, see Create and manage network attachments.

  • To save the trigger configuration, click Save trigger.
    • Note: Saving your {{eventarc_name_short}} trigger configuration defines the rules that determine when and how {{eventarc_name_short}} responds to events. This configuration is stored as part of your trigger definition in {{integration}}. However, these rules are not immediately applied. You must publish the integration to deploy these rules and activate the {{eventarc_name_short}} trigger. Until published, the trigger remains inactive and doesn't process events. For information about publishing integrations, see Publish integrations.

Create an {{eventarc_name_short}} trigger using the Eventarc console

To create an {{eventarc_name_short}} trigger from the {{eventarc_name_short}} console page, do the following steps:

  1. In the {{console_name_short}}, go to the {{eventarc_name_short}} {{and_then}} Triggers page. {% setvar button_text %}Go to Triggers{% endsetvar %} {% setvar console_path %}eventarc/triggers{% endsetvar %} {% include "docs/includes/_open_console.html" %}

    The Triggers page lists all the triggers in your project.

  2. Click add_box Create trigger.
  3. In the Create trigger page, provide the following inputs:
    1. Trigger name: enter a name for the trigger.
    2. Trigger type: select one of the following trigger types:
      • Google sources: select this option to create a trigger for {{pubsub_name_short}}, {{storage_name}}, {{firestore_name}}, and other Google event providers.
      • Custom: select this option to create a trigger for a custom event source.
      • Third-party: select this option to create a trigger for non-Google providers that offer an Eventarc source. For more information, see Third-party events in Eventarc.
    3. Event provider: select the event provider for the trigger. The list of event providers depends on the trigger type. For more information, see Event providers and destinations.
    4. Event type: select the event type for the trigger. Your trigger configuration varies based on the event provider you select. For more information, see Event types.
    5. Event data content type: select the data type for the event content. Select either application/json or application/protobuf.
    6. Region: select the region for the trigger.
      Tip: For optimal performance and to minimize latency, ensure that your {{eventarc_name_short}} trigger, the monitored Google Cloud resource, and your service are all located in the same {{gcp_name_short}} region.
    7. Service account: select the service account to use for the trigger. If you don't select a service account, the trigger uses the {{compute_name_short}} default service account as its identity when invoking your service. The service account associated with the trigger must have the required permissions to invoke your service.
    8. Event destination: select Application Integration.
    9. Select an Integration: select the integration for which you want to create the trigger. If the integration is published, {{integration}} creates a new draft version of the integration to add the {{eventarc_name_short}} trigger to.
    10. Network attachment: select the network attachment to use for the trigger. This list is generated based on the network attachments created in your project.

      Network attachments facilitate communication between service producer {{vpc_name_first_mention}} networks and consumer {{vpc_name_short}} networks. {{eventarc_name_short}} uses the configured network attachment to establish a connection to an internal HTTP endpoint hosted within the consumer's {{vpc_name_short}} network, enabling delivery of event data. For more information about network attachments, see About network attachments.

  4. Click Open integration.

    The {{integration}} editor page opens with the integration selected and the new {{eventarc_name_short}} trigger element added to the integration.

  5. Click the {{eventarc_name_short}} trigger element and open the trigger configuration pane.
  6. Review and verify the trigger configuration in the {{eventarc_name_short}} trigger editor page.
  7. Connect the new {{eventarc_name_short}} trigger to your integration flow and publish the integration.
    8. Note: You must publish the integration to deploy and activate the {{eventarc_name_short}} trigger. Until published, the trigger remains inactive and will not process events. For information about publishing integrations, see Test and publish integrations.
