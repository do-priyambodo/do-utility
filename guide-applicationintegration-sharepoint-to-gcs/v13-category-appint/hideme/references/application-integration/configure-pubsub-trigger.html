The {{apip_trigger_pubsub}} lets you run your integration based on the events in your {{pubsub_name}} middleware.

Using this trigger, you can execute an integration whenever a publisher writes a message to your chosen topic. The topic can exist in the same {{gcp_name}} project as the integration, or in a different {{gcp_name}} project. When this trigger executes, it also writes the published message to the CloudPubSubMessage output variable. Based on your requirements, you can use this variable in your downstream triggers or tasks.

To learn how to create a sample integration using {{integration}} and the {{apip_trigger_pubsub}} trigger, see Listen to Cloud Pub/Sub topic and send an email.

Before you begin

Ensure that you perform the following tasks in your {{gcp_name}} project before configuring the {{apip_trigger_pubsub}}.

  • Assign the Project IAM Admin (roles/resourcemanager.projectIamAdmin) to the user creating this trigger.
  • Enable the {{pubsub_name}} API for the {{gcp_name}} project that has your integration.

    • If the {{pubsub_name}} API is protected by a service perimeter, ensure that you define the following IAM roles to manage these service perimeters:

    • Access Context Manager Admin (roles/accesscontextmanager.policyAdmin)
    • Access Context Manager Editor (roles/accesscontextmanager.policyEditor)
    • Access Context Manager Reader (roles/accesscontextmanager.policyReader)

  • Confirm that you have a {{pubsub_name}} topic. To create a new {{pubsub_name}} topic:
    1. Go to the {{pubsub_name}} topics page in the Cloud console.

      Go to the {{pubsub_name}} topics page

    2. Click Create a topic.
    3. In the Topic ID field, provide a unique topic name.
    4. Click Save.
    Note: You don't have to create a subscription for the trigger to work. {{product_name}} automatically creates a subscription for a topic when you publish your integration, and you can view all your subscriptions in the {{pubsub_name}} subscription page.

    Starting with the {{product_name}} December 01 2023 release, you will be billed for your subscriptions as per the {{pubsub_name}} billing rates. Subscriptions for your existing {{apip_trigger_pubsub}} will continue to work as before, and they aren't billed. However, if you update the topic of an existing {{apip_trigger_pubsub}}, {{pubsub_name}} billing will be applied to the updated {{apip_trigger_pubsub}}.

For more information about managing access using IAM roles, see Access control.

For information on granting roles to principals, see Granting, changing, and revoking access.

Configure a {{apip_trigger_pubsub}}

Click the {{apip_trigger_pubsub}} element in the integration editor to view the trigger configuration pane, and then do the following tasks:

  1. Specify the topic the trigger should listen to in the {{pubsub_name}} topic field in the following format: 
    projects/PROJECT_ID/topics/TOPIC_ID

    To view all the available topics in your {{gcp_name}} project, see the Topic name column in Topics. You can also select a topic from a different {{gcp_name}} project.

    Replace the following:
    • PROJECT_ID: the {{gcp_name}} project where your topic is created.
    • TOPIC_ID: the ID of your {{pubsub_name}} topic. The topic can be in the same {{gcp_name}} project as that of your integration, or can be present in a different {{gcp_name}} project.
  2. Select the service account which you want to use to invoke the integration. You can also choose a config variable that you have created in the integration. The value of the config variable must be a service account email address with the following format: SERVICE_ACCOUNT_NAME@project-id.{% dynamic print variables.domains.sa_iam_domain %}. For example, application-integration-invoke@project123.{% dynamic print variables.domains.sa_iam_domain %}.

    For information about how to create a config variable, see create a variable, and about how to create a service account, see Creating a service account.

    If you choose a config variable, you must grant the required roles to the service account that the config variable refers to. For more information about how to grant roles, see Required IAM roles.

    If there are no service accounts, create a service account by clicking Create new in the Service account list.

    Note: Starting with the {{product_name}} December 01 2023 release, when you configure the {{apip_trigger_pubsub}}, you must specify a service account for the trigger. Your existing {{pubsub_name}} triggers, that don't have any service account associated with them, will continue to work as before. However, if you want to modify any of the existing {{pubsub_name}} triggers, you must configure a service account to continue using those triggers.

By default, the expiration period option for subscriptions is set to never expire. If you want to set the expiration to another value, in the {{console_name}}, go to the {{pubsub_name}} subscription page and edit the subscription.

Trigger Output

For each event, the {{apip_trigger_pubsub}} generates a CloudPubSubMessage output variable which you can use in your downstream tasks. The variable is in JSON format and has the PubsubMessage structure.

Create subscription with filter for your integration

If you want to set a filter for the {{pubsub_name}} subscription to process specific events in your integration, follow these steps:

  1. Publish the integration with the {{apip_trigger_pubsub}} configured. This step creates the subscription in the integration project.
  2. In the {{console_name_short}}, go to the Subscriptions page.

    Go to Subscriptions

  3. Open the subscription and copy all the details including the subscription name.
  4. Delete the existing subscription.
  5. Create a new subscription with all the details of the deleted subscription–including the subscription name–that you copied in step 3 and also add a filter in the new subscription.

    The newly created subscription triggers the integration with filtered messages.

SLA exclusions

The {{apip_trigger_pubsub}} is dependent on the {{pubsub_name}} product. As this dependency is external to the {{product_name}}, the {{product_name}} Service Level Agreement (SLA) terms and conditions are applicable only when the {{apip_trigger_pubsub}} successfully starts the execution of an active integration.

Use {{pubsub_name}} triggers with {{vpcsvcctl_name}}

When you configure a {{pubsub_name}} trigger in {{integration}} and attempt to publish the integration, the creation of the required {{pubsub_name}} subscription fails if the {{pubsub_name}} API is protected by a service perimeter in {{vpcsvcctl_name}}.

Service perimeters enforce security boundaries around your {{gcp_name}} resources, and these policies can prevent {{integration}} from creating the necessary {{pubsub_name}} subscription during integration publishing.

To successfully configure and publish a {{pubsub_name}} trigger in these scenarios, perform the following steps:

  1. Temporarily remove the {{pubsub_name}} API from the service perimeter:
    1. In the {{gcp_name}} console, go to the {{vpcsvcctl_name}} page.

      Go to VPC Service Controls

    2. Select the {{gcp_name}} project that contains the {{pubsub_name}} topic to be used in the integration.
    3. (Optional) If no access policy is configured for the selected project, the message No VPC Service Perimeters found in current scope will appear. In this case, click Switch to organization scope to navigate to the organization level.
    4. On the {{vpcsvcctl_name}} page, in the table, click the name of the service perimeter that you want to modify.
    5. On the Service perimeter details page, click Edit.
    6. In the Restricted services pane, filter and select the pubsub.googleapis.com service.
    7. Click delete Remove Selected.
    8. Click Save. In the confirmation dialog, click Confirm to remove the {{pubsub_name}} API from your {{vpcsvcctl_name}} service perimeter.
  2. Add and configure the {{pubsub_name}} trigger in your integration.
  3. Publish the integration.
  4. Add the {{pubsub_name}} API back to the service perimeter:
    1. Go to the {{vpcsvcctl_name}} page.

      Go to VPC Service Controls

    2. On the {{vpcsvcctl_name}} page, in the table, click the same service perimeter that you modified earlier.
    3. On the Service perimeter details page, click Edit.
    4. In the Restricted services pane, click add Add services.
    5. Filter and select the pubsub.googleapis.com service.
    6. Click Add selected services.
    7. Click Save. In the confirmation dialog, click Confirm to add the {{pubsub_name}} API back to your {{vpcsvcctl_name}} service perimeter.

Considerations

  • The maximum size of a {{pubsub_name}} message is 10 MB. To understand all the limitations of {{pubsub_name}}, see {{pubsub_name}} quotas and limits.
  • It is possible that {{pubsub_name}} writes the same message more than once to a topic. To understand this behavior of {{pubsub_name}}, see At-Least-Once delivery. Therefore, you must design your integration to be idempotent so that it doesn't allow a message to be written more than once to a topic.

    The message_id field in the message header uniquely identifies a {{pubsub_name}} message.

  • {{product_name}} now lets you publish messages to {{pubsub_name}} topics. You can use ordering keys to achieve First-In, First-Out (FIFO) message processing. When you set an ordering key in the Publish Pub/Sub Message task, messages are received in the correct order. This improves reliability for integrations that need ordered message processing. To learn more about how to use ordering keys when you publish messages, see Using ordering keys.
  • If you unpublish an integration, the {{pubsub_name}} subscription is not deleted to prevent loss of messages. However, if you delete the published version or all the integration versions of an integration, then the {{pubsub_name}} subscription is also deleted.
